# インフルエンサーAIチャットボット 汎用設計書

どんなインフルエンサー（タレント・VTuber・配信者・キャラクター）でも、同じ構造でファンメイドAIチャットボットを作るための設計書です。実装例として本リポジトリの `shokotan-ai/`（しょこたんAI）を参照してください。

---

## 1. 全体アーキテクチャ

```
┌─────────────────────────────────────────────┐
│  index.html（1ファイル完結・サーバー不要）      │
│                                             │
│  ┌───────────┐  ┌────────────────────────┐  │
│  │ UIテーマ層 │  │ 応答エンジン             │  │
│  │ CSS変数で │  │  ①ナレッジ検索(強一致)    │  │
│  │ 色を差替え │  │  ②ペルソナルール(正規表現) │  │
│  └───────────┘  │  ③ナレッジ検索(弱一致)    │  │
│  ┌───────────┐  │  ④フォールバック(口調維持) │  │
│  │ アバター層 │  └────────────────────────┘  │
│  │ 3段ﾌｫｰﾙﾊﾞｯｸ│  ┌────────────────────────┐  │
│  └───────────┘  │ knowledge.js（任意・分離）│  │
│                 └────────────────────────┘  │
└─────────────────────────────────────────────┘
        ↓ git push
   GitHub Pages（無料ホスティング）
```

設計原則:
- **静的ファイルのみ**（HTML+CSS+JS）。APIキー・サーバー・ビルド不要なので、誰でもフォークして差し替えられる
- **個性は全部「データ」に寄せる**。コードは共通、差し替えるのは後述の5つのカスタマイズポイントだけ

## 2. カスタマイズポイント（個性の差し替え箇所）

| # | レイヤー | 何を差し替えるか | 実装上の場所（しょこたんAIの例） |
|---|---|---|---|
| 1 | ペルソナ定義 | 性格・好きなもの・背景設定 | `PERSONA.md`（設計資料。コードには直接入らない） |
| 2 | 口調エンジン | 一人称・語尾・口癖・挨拶・感嘆詞 | `RULES` / `FALLBACK` / `KNOW_PRE` / `KNOW_POST` の文字列 |
| 3 | UIテーマ | イメージカラー2色＋装飾モチーフ | CSS `:root` の変数（`--sky`系・`--pink`系）と背景装飾 |
| 4 | アバター | 本人画像（権利クリア必須）＋予備SVG | `PHOTO_LOCAL` / `PHOTO_COMMONS` / `AVATAR_SVG` |
| 5 | ナレッジ | 本人の活動・作品・発言に関する知識 | `knowledge.js` の `KNOWLEDGE` 配列 |

## 3. ペルソナ定義仕様（PERSONA.md）

コードを書く前に必ず作る。最低限の項目:

```markdown
## ①性格
- 核となる性格3〜5点（例: ハイテンション/オタク気質/猫好き）
- 大切にしている価値観・座右の銘
- 得意分野・代表作

## ②口調
| 要素 | 内容 |
|---|---|
| 一人称 | （例: しょこたん / 俺 / わたくし） |
| 語尾・活用 | （例: 〜ヲ / 〜なのだ / 〜っす） |
| 挨拶 | （例: トゥットゥルー♪） |
| 強調表現 | （例: ギザ〜、ギガント） |
| 口癖・定番フレーズ | 5〜10個 |
| 文体の特徴 | 感嘆符の量、絵文字・顔文字、ネットスラング |

## 名言集
- 出典が確認できるものだけ。捏造しない

## 権利関係メモ
- 使用画像の出典・ライセンス・クレジット表記
```

## 4. 応答エンジン仕様

### 4.1 優先順位（reply関数）

```
入力テキスト
 → ナレッジ強一致（スコア≧3）       … 専門知識の質問
 → ペルソナルール（正規表現マッチ）   … 雑談・定番ネタ
 → ナレッジ弱一致（スコア≧1）       … 知識っぽい質問の拾い漏れ
 → フォールバック                  … 口調を維持した相槌
```

雑談（猫・挨拶など）がナレッジに食われないよう、強一致のしきい値は3を推奨。

### 4.2 ペルソナルール（RULES）

```js
{ re: /(キーワード正規表現)/, res: ['返答候補1', '返答候補2'] }
```

- 最低限入れるカテゴリ: 挨拶 / 自己紹介（誰？） / 本人の代表的な好きなもの2〜3分野 / 褒め言葉への反応 / 励まし（疲れた・悲しい） / 名言リクエスト / 感謝 / 別れ際
- **励まし系は最重要**。ユーザーが一番話しかける内容なので、本人らしい優しさが出る返答を厚めに
- `pick()` は直前と同じ候補を避ける（連続同文の防止）

### 4.3 ナレッジベース（knowledge.js）

```js
const KNOWLEDGE = [
  { "topic": "見出し",
    "keywords": ["検索語1", "口語的な聞き方も入れる"],
    "answer": "2〜4文の要約回答。" },
];
```

- スコアリング: キーワード包含で加点（4文字以上は2点、それ未満は1点）。最高得点のエントリを採用
- keywordsには**ユーザーの口語**（「どっちが得？」「いくら必要？」）を必ず混ぜる
- answerは要約に徹する。**原典の長文転載は権利・分量の両面でNG**
- ナレッジ回答は `KNOW_PRE`（前置き）+ answer + `KNOW_POST`（締め）で挟み、口調を本人化する

## 5. UIテーマ仕様

- 色は CSS `:root` 変数に集約。**メインカラー系統5変数＋サブカラー系統5変数**を差し替えるだけで全体が変わる
- 背景はメイン→サブの斜めグラデーション＋装飾（雲/星/ハート等、本人のモチーフに合わせる）
- チャットUI構成: ヘッダー（アバター＋名前＋オンライン表示）/ プロフィールカード / 吹き出し（AI=サブ色、ユーザー=メイン色）/ クイック返信チップ / 入力バー / フッター（免責＋画像クレジット）

### モバイル必須対応（ハマりどころ）
- 高さは `vh` でなく **`dvh`**（アドレスバー分の見切れ防止）
- 600px以下は全画面レイアウト
- 入力欄に `flex: 1 1 0; min-width: 0;`（送信ボタンの横見切れ防止）
- 入力欄フォントは**16px以上**＋viewportに `maximum-scale=1.0`（iOS自動ズーム防止）
- `env(safe-area-inset-bottom)` でホームバー回避

## 6. 画像・権利仕様（最重要）

実在の人物の画像・人格には権利があります。**ここを飛ばすと公開できません。**

| 項目 | ルール |
|---|---|
| 本人写真 | Web上の写真の無断使用はNG（肖像権・パブリシティ権・撮影者の著作権）。使えるのは ①Wikimedia Commons等のフリーライセンス画像（CC BY等、クレジット表記必須）②本人・事務所の許諾を得た画像 ③自作のオリジナルイラストのみ |
| クレジット | CC BYなら「撮影者名＋出典リンク＋ライセンスリンク」をフッターに常時表示 |
| 非公式表記 | 「ファンメイドのパロディであり本人・所属事務所とは無関係」をフッターに明記 |
| 名言 | 出典確認できるもののみ。歌詞の引用はJASRAC等の管理対象なのでNG |
| 表示の堅牢化 | 3段フォールバック: ①リポジトリ内画像 → ②出典元ホットリンク → ③オリジナルSVGアバター |
| 商用利用 | この構成はあくまで非営利ファン活動の範囲。収益化するなら事務所の許諾が必須 |

補足: 開発環境がネットワーク制限下にある場合、リポジトリのGitHub Actionsで画像を取得してコミットする方法が使える（`.github/workflows/fetch-shokotan-photo.yml` 参照）。

## 7. デプロイ仕様

- GitHub Pages（mainブランチ配信）に `<ボット名>/` ディレクトリとして配置
- 1リポジトリに複数ボットを並べられる（`/shokotan-ai/`、`/xxx-ai/`…）
- 公開URL: `https://<user>.github.io/<repo>/<bot-dir>/`

## 8. テスト仕様

Playwright等で最低限以下を自動確認:
1. 知識質問→ナレッジ回答が返る / 雑談→ペルソナルールが返る（reply()を直接呼んで検証）
2. スマホビューポート（390px・320px）で入力欄と送信ボタンが画面内に収まる
3. 会話スクリーンショットの撮影（成果物として保存）

## 9. 将来拡張

- **Claude API化**: ルールベースをやめ、PERSONA.md をシステムプロンプトに、knowledge.js をRAG資料にしてClaude APIへ接続すると自然会話になる。ただしAPIキーを静的ページに置けないため、小さなプロキシサーバー（Cloudflare Workers等）が必要
- ナレッジの追記は knowledge.js に配列要素を足すだけ（コード変更不要）
