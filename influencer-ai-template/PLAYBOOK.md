# インフルエンサーAIチャットボット 制作手順書

設計の考え方は `DESIGN.md` を参照。ここは「作業手順」だけをまとめたものです。
**Claude Codeに丸投げする場合は、最後の「Claude Code用プロンプト」をコピペすればSTEP 1〜7を自動でやってくれます。**

---

## STEP 0｜企画と権利確認（人間がやる・必須）

- [ ] 対象インフルエンサーを決める
- [ ] **画像の入手ルートを確認**（下のどれかに該当しなければアバターは自作イラスト一択）
  - Wikimedia Commons で本人を検索 → フリーライセンス（CC BY等）の写真があるか
  - 本人・事務所の許諾が取れるか
- [ ] 非営利のファン活動の範囲か確認（収益化するなら許諾必須）

## STEP 1｜リサーチ（30分〜1時間）

集めるもの（出典URLをメモしながら）:
1. 性格・人柄がわかる記事/インタビュー 3本以上
2. 口調の特徴（一人称・語尾・口癖・挨拶・独特の語彙）→ ファンwiki・大百科系サイトが早い
3. 名言・座右の銘（出典が確認できるものだけ）
4. 代表作・活動内容・好きなもの
5. イメージカラー（公式カラー、メンバーカラー、衣装・ロゴの色）

## STEP 2｜PERSONA.md を書く（30分）

`shokotan-ai/PERSONA.md` をテンプレートに、①性格 ②口調表 ③名言集 ④権利関係メモ を埋める。
**口調表が品質の8割を決める**ので、語尾・強調語・口癖は実例ごと書き写す。

## STEP 3｜アプリ本体を作る（1〜2時間）

1. `shokotan-ai/index.html` を `<新ボット名>/index.html` にコピー
2. 差し替え（DESIGN.md「カスタマイズポイント」の5箇所）:
   - [ ] `:root` のCSS変数をイメージカラー2系統に
   - [ ] 背景装飾のモチーフ（雲/星/音符/炎…本人に合わせる）
   - [ ] タイトル・ヘッダー名・プレースホルダー文言
   - [ ] `RULES` / `FALLBACK` / `QUOTES` を新しい口調・ネタで書き直す（カテゴリ構成は流用）
   - [ ] `AVATAR_SVG`（予備アバター）と `PHOTO_LOCAL` / `PHOTO_COMMONS`
   - [ ] フッターの非公式表記＋画像クレジット
3. クイック返信チップを本人の鉄板ネタ4〜5個に変更

## STEP 4｜画像を入れる（30分）

1. フリーライセンス画像の**正確なファイル名とライセンス・撮影者**を確認
2. 画像をリポジトリに入れる（`<ボット名>/avatar.jpg`、幅600px程度に縮小）
   - 開発環境から直接ダウンロードできない場合: `.github/workflows/fetch-shokotan-photo.yml` を複製し、URLと出力先を書き換えてpush（CIが取得してコミットしてくれる）
3. フッターにクレジット表記（撮影者名＋Commonsリンク＋CC BYリンク）

## STEP 5｜ナレッジを入れる（任意・30分〜）

本人の活動知識や、載せたい資料がある場合:
1. `knowledge.js` を作る（形式は `{topic, keywords[], answer}` の配列）
2. 資料が長文の場合は**要約**してエントリ化（全文転載しない）
3. keywordsに口語的な聞き方を混ぜる
4. `index.html` 側は `<script src="knowledge.js">` と検索ロジックをそのまま流用

## STEP 6｜テスト（15分）

- [ ] 挨拶・雑談・励まし・名言・知識質問の5パターンに「らしい」返答が返る
- [ ] スマホ幅390px・320pxで送信ボタンが見切れない
- [ ] 画像が出ない環境で予備SVGに切り替わる（shokotan.jpgをリネームして確認）
- [ ] 会話スクリーンショットを撮って `<ボット名>/screenshot.png` に保存

## STEP 7｜デプロイ（5分）

```bash
git add <ボット名>/
git commit -m "<ボット名>AIチャットボットを追加"
git push
# GitHub Pages有効なリポジトリならmainにマージで公開される
```

公開URL: `https://<user>.github.io/<repo>/<ボット名>/`

### 運用Tips
- **ナレッジ追加だけしたい場合**: 「`<ボット名>/knowledge.js` に以下の内容をエントリ追加して。形式は既存に合わせ、keywordsに口語の聞き方も入れて」と資料を渡す
- **口調の微調整**: 「RULESとFALLBACKの返答をもっと【特徴】っぽくして。例: 【実際の発言例】」のように実例を渡すと精度が上がる
- 大量の資料を読ませる場合はサブエージェント（Task/Agent）に要約させると速い
